Build a reviewer cockpit, not a label-upload toy.

V1 is a stateless spirits-label discrepancy review session: enter application facts, upload/paste 1–25 labels, extract visible evidence, run deterministic rules with citations, then let a human disposition the case and export the packet.

Architecture

Keep the good kernel: model extracts, deterministic rules decide, human owns final disposition.

Runtime modules

  • schemas.ts: request/response contracts
  • extract.ts: text fallback + guarded vision extraction
  • ruleProfiles.ts: spirits/wine/malt requirement sets
  • rules.ts: deterministic checks
  • batch.ts: stable IDs + per-label statuses
  • exportPacket.ts: JSON + Teams-style summary

UI components

Build sequence

TDD-first. If a step changes behavior and has no failing test first, it is wrong.

Rule contract tests

Failing tests for requirement refs, severity, low-confidence gating, exact warning handling, and spirits ABV mismatch.

API contract tests

Failing tests for label IDs, per-label failure isolation, malformed model output, validation errors, and response shape.

Refactor kernel

Extract schemas/extractor/profiles/export modules without changing behavior; keep tests green.

Add v1 response shape

Return batchId, labelId, requirementRef, severity, confidence, and sanitized failures.

Replace page with cockpit layout

Split current monolith into panels: application, queue, evidence, findings, disposition/export.

Verify visually and functionally

Run tests/lint/build, open browser, submit demo batch, check console, export JSON/copy summary.

Risks and hard rules

These prevent the product from lying.

Hard rules

  • Never call it approval. Call it review recommendation.
  • Low confidence cannot become clean pass.
  • Every failed/warning check has a source ref.
  • Layout compliance is “unverified” unless image analysis proves it.
  • Model output is evidence extraction only.

Known risks

  • OCR can miss curved/glossy labels.
  • Regulatory commodity boundaries are tricky.
  • TTB/COLA integration can become procurement swamp.
  • Batch scale claims are fake without persistence/queue infra.
  • Reviewer trust dies if rationale is vague.

Definition of done for V1

The app is “built” only when a reviewer can complete this loop without guessing what to do next.

  • Enter application record.
  • Submit 1–25 labels/texts.
  • See queue status per label.
  • Select a label and inspect image/OCR evidence.
  • See findings with expected/observed/rationale/source refs.
  • Make disposition: accept, request correction, override with reason, or SME review.
  • Export JSON packet and copy Teams-style summary.
  • Tests/lint/build pass.

Non-goal: persistent case management. If you refresh, the session can vanish. That is acceptable for V1.
