Presearch summary

Source artifact: docs/PRESEARCH.md. Core conclusion: use a multimodal extractor only for evidence extraction, then deterministic rules for compliance checks. Receipts over magic.

Confirmed requirements

Standalone proof-of-concept

  • Upload label artwork and compare visible fields against application data.
  • Core checks: brand, class/type, ABV/proof, net contents, bottler/importer/origin where applicable, Government Health Warning.
  • Keep mixed technical-comfort users from hunting for buttons.
Trust posture

Review assistant, not approver

  • Return decision, score, evidence cards, expected vs observed, and rationale.
  • Use visible reasoning to earn reviewer trust.
  • Do not claim layout/font compliance from plain text extraction.
Production caveats

Government deployment is mostly risk surface

  • Retention, audit logs, RBAC, approved model path, and threat model are required later.
  • FedRAMP/ATO path matters more than UI polish.
  • False approvals are more damaging than extra needs-review flags.

Architecture

Current architecture is deliberately small: Next.js UI, server route, optional vision extractor, deterministic rules, no persistence by default. Future government/Teams paths sit behind gates.

Blue: reviewer-facingGreen: API orchestrationRed: compliance rulesDashed: optional/future path

Reviewer workflow

The closed loop matters: intake → extraction → deterministic checks → human disposition → export/escalation. If the loop cannot close, don’t build broad public UI yet.

Requirements board

Filter the next-phase requirements by implementation track. These are pulled from the spec and converted into a buildable board.

Pilot

Batch review is credible

Return one result per label, isolate per-label extraction failures, show progress/result states, and keep 1-label and 25-label cases readable.

Pilot

Evidence-first trust

Every check shows expected value, observed value, rationale, requirement reference, severity, and confidence/evidence where available.

TDD/API

Pure rule tests first

Normalization, commodity required fields, warning exactness, optional vs required fields, low-confidence behavior, and decision aggregation start as failing tests.

TDD/API

Route contract tests

Validate accepted payloads, structured validation failures, max batch limits, bad MIME/type handling, extractor failure, and text-only fallback.

Security

No accidental persistence

Raw uploads, model responses, review packets, and data URLs are not stored server-side until retention, encryption, audit, and deletion are designed.

Security

Government deployment assumptions

Approved model provider path, FedRAMP/ATO review, sanitized logs/errors, payload limits, rate limiting, and human-final-authority copy.

Teams/Internal

Copy/export now

Reviewers can download a review packet or copy a Teams-ready summary block with batch ID, decision, warnings/fails, disposition, and notes.

Teams/Internal

Integration later

/api/integrations/teams/summary stays disabled by default behind tenant-approved credentials and sends summary metadata, not raw images.

Office-hours options

The gstack office-hours conclusion: start with a narrow shadow pilot if agency access exists. If not, learn faster through industry pre-submission checking. Full federal integration is a later gate, not the starting line.

Best if we have reviewer/sponsor access

Run a non-authoritative pilot with 50–100 historical/anonymized labels, measure time saved and false-pass rate, and keep the human reviewer in control.

TDD-first implementation sequence

Next implementation should dispatch subagents by track, but every functional change begins with a failing test. No production code first. Boring, reliable, correct.
